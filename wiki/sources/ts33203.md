# 3GPP TS 33.203 v14.1.0 — Access Security for IP-based Services

## Scope

Specifies security features and mechanisms for secure access to the IMS for 3G mobile telecommunications. Covers only the UE-to-IMS security — specifically how SIP signalling is protected between subscriber and IMS, how the subscriber is authenticated, and how the subscriber authenticates the IMS. Also applicable (via normative annexes) to fixed broadband, 3GPP2, non-3GPP WLAN, and WebRTC access.

## Chunk Log

### Chunk ts33203-1: §4–§9 (Main Body)

**Ingested:** 2026-04-12  
**Sections:** §4 (security architecture), §5 (features), §6 (mechanisms), §7 (SA setup procedure), §8 (ISIM), §9 (IMC)  
**Pages produced:**
- `wiki/concepts/IMS-access-security.md` — full architecture + features + mechanisms + ISIM/IMC
- `wiki/procedures/IMS-AKA-registration.md` — step-by-step IMS AKA + IPsec SA setup flow

**Key findings:**
- 5 security associations defined in IMS; only SA ② (Gm: UE↔P-CSCF) is specified in this document
- IMS AKA is independent of EPS/UMTS AKA — even shared-key UICC runs separate auth producing different session keys per domain
- P-CSCF is the IPsec termination anchor: strips CK/IK from challenge before forwarding to UE; derives IK_ESP/CK_ESP; creates 4 SAs
- 4-port model (port_us/uc/ps/pc) plus unique SPIs prevents reflection attacks
- Integrity mandatory; confidentiality operator-optional (NULL cipher allowed)
- Topology hiding uses AES-CBC with random IV — same plaintext never produces same ciphertext
- ISIM on UICC takes priority over USIM; session keys deleted at power-off

### Chunk ts33203-2: Annexes H + I + M + N + O + P

**Ingested:** 2026-04-12  
**Sections:** Annex H (RFC 3329 SIP Security Mechanism Agreement BNF), Annex I (key expansion: IK_ESP/CK_ESP), Annex M (NAT traversal: UDP tunnel mode, port 4500), Annex N (SIP Digest: SD-AV, HTTP Digest flow, dynamic password change), Annex O (TLS: profile, session setup during/prior-to registration, certificate validation), Annex P (auth scheme coexistence: 6 schemes, P-CSCF/S-CSCF selection rules)  
**Pages produced:**
- `wiki/concepts/IMS-auth-alternatives.md` — Annex H (BNF), Annex I (key tables), Annex M (NAT), Annex N (SIP Digest flow), Annex O (TLS), Annex P (coexistence), auth scheme comparison table

**Key findings:**
- SIP Digest is fundamentally incompatible with IPsec — no session keys generated; combined use is technically impossible
- SIP Digest + TLS is the non-3GPP equivalent of IMS AKA + IPsec — provides equivalent security properties via different mechanisms
- AES-GCM and AES-GMAC require 32-bit salts derived from CK_IM||IK_IM via KDF (TS 33.220 Annex B); simple algorithms (AES-CBC, HMAC-SHA-1-96) use CK_IM/IK_IM directly or with zero-padding
- NAT traversal uses UDP port 4500 encapsulation (RFC 3948); P-CSCF detects NAT by comparing packet source IP with Via header IP
- "ipsec-3gpp" extends RFC 3329 with bidding-down protection: server stores Security-Client from SM1; client echoes it in SM7; server verifies consistency
- Auth scheme coexistence (Annex P) requires same S-CSCF to serve fixed and mobile subscribers — multi-scheme capability mandatory
