# Source: 3GPP TS 23.214 v16.1.0

**Full title:** Architecture enhancements for control and user plane separation of EPC nodes
**Version:** 16.1.0 (Release 16, 2020-07)
**File:** `raw/papers/ts_123214v160100p.pdf`
**Total pages:** ~94

## Scope

Stage 2 specification for CUPS (Control and User Plane Separation) of EPC's SGW, PGW, and TDF. Defines the Sx reference point family (Sxa/Sxb/Sxc) between CP and UP functions. Covers architecture model, functional splits, Sx session/reporting/management procedures, and all Sx parameter tables (PDR, URR, FAR, QER).

## Key Findings (ts23214-1)

- **CUPS is fully transparent to UE and RAN:** the split does not require any changes to eNodeB, RNC, or UE. The MME sees the SGW-C partition as a legacy SGW.
- **S2a/S2b/S5/S8 restriction:** CUPS only supported with GTP-based variants; PMIP and S2c not supported.
- **DPI moves to UP function:** service detection (deep packet inspection) is performed by SGW-U / PGW-U / TDF-U. The CP function never processes user data.
- **Bearer binding split:** PGW-C makes the binding _decision_ (it knows the QoS/TFT policy from PCRF); PGW-U performs binding _verification_ and DL traffic mapping. This preserves PCC centralization while offloading forwarding.
- **TDF APN-AMBR scope narrower than PGW:** TDF-U APN-AMBR covers only flows in the current TDF session — not the full APN-AMBR across all PDN connections.
- **SGW-C Partitioning enables geo-distributed SGW-U:** one SGW-C can be logically split into partitions each paired with a geographically scoped SGW-U, enabling RAN-site-local user plane without requiring per-RAN-site control plane nodes.

## Key Findings (ts23214-2)

- **Charging interface termination never moves to UP:** Gx/Sd/Gy/Gz/OFCS all terminate at CP function; UP only provides raw usage counts via URRs. OCS/OFCS/PCRF see no change from non-split architecture.
- **F-TEIDu allocation is UP-function mandatory:** UP manages its own TEID space; CP never allocates TEIDs itself. This decouples TEID namespace from control-plane identity.
- **Buffering split is configurable per UE session:** SGW-C decides where to buffer based on UE capability and local policy. SGW-U buffering is mandatory; SGW-C buffering is optional except for NB-IoT/eDRX UEs.
- **End marker construction is configurable:** either CP or UP can construct end markers — network configuration chooses. The two modes have different Sx signalling flows but identical outcome (end markers sent on old path after last PDU).
- **PGW-U must enforce APN-AMBR across all IP-CAN sessions:** PGW-C provides APN-AMBR + QER correlation ID. ARP is never sent to UP (admission control remains CP-only).
- **UP function selection uses a capability-matching model:** CP dynamically learns UP load via Sx Load Control and matches required features (DPI capability, NR dual connectivity support) to UP capabilities declared at Sx setup. Same PGW-U must be selected for same UE+APN when APN-AMBR enforcement is required.
- **PFD conflict risk with multi-CP/single-UP:** SDCI rules warn that if multiple CP functions control a single UP and provide different PFDs for the same App ID, the latest received wins. This is an operational constraint on CUPS deployments.

## Key Findings (ts23214-5)

- **Sx Association is the prerequisite for all Sx sessions:** CP and UP must establish an association (CP or UP can initiate; CP must support UP-initiated; CP must only associate with a UP that supports F-TEID allocation) before any Sx session establishment can occur.
- **Association Release is CP-only:** Only the CP function may initiate Sx Association Release (OAM reason); UP cannot unilaterally tear down the association.
- **Bearer ID never travels over Sx:** CP maps EPS Bearer ID to Sx rule IDs (FAR/URR/QER IDs in PDRs) internally. The UP function operates purely on rule IDs and never sees a bearer ID — this is the key abstraction that makes CP/UP separation clean.
- **APN-AMBR correlation is by QER correlation ID:** CP provides the same QER (same correlation ID) across all non-GBR PDRs of all PDN connections to the same APN, enabling PGW-U to enforce APN-AMBR across multiple simultaneous PDN connections.
- **One URR per Charging Key and/or Monitoring Key:** CP maintains the mapping; may combine a Charging + Monitoring Key into one URR if their requirements fully overlap. PDR references the applicable URR IDs.
- **PFD provisioning replaces all existing PFDs for an App ID:** Sending PFD(s) + Application ID replaces the entire PFD set; sending Application ID alone removes all PFDs. Removal semantics are atomic per App ID.
- **Network instance in PDR/FAR is for IP domain disambiguation:** Required when UP handles multiple APNs with overlapping IP addresses or connects to peers in different IP routing domains.

## Key Findings (ts23214-4)

- **PCC interactions are PGW-C→PGW-U modifications only:** PCRF (Gx) rule changes never touch UP directly; PGW-C translates PCRF decisions into updated PDRs/FARs/URRs via Sx Modification.
- **TDF solicited session is symmetric:** TDF-C establishes and terminates the TDF Sx session in lockstep with the PCRF-indicated TDF session lifecycle on Sd.
- **Gn/Gp SGSN bypasses SGW entirely for PDN connection:** PGW-C↔PGW-U Sx is the only interaction; no SGW-C/SGW-U Sx exists in this path.
- **S4 SGSN follows the same 2-session pattern as E-UTRAN:** both SGW-C↔SGW-U and PGW-C↔PGW-U Sx sessions created at PDN establishment; both terminated at deactivation.
- **Sx reporting is always UP-initiated, but trigger-configured by CP:** CP installs trigger conditions (URR thresholds, traffic detection requests) during Sx session management; UP fires reports autonomously when those conditions are met.
- **Four session-level report triggers:** usage (all UP types), start/stop of traffic detection (PGW-U/TDF-U), and first DL for idle-mode UE (SGW-U only).
- **Sx Report ACK can carry buffering extension params:** if a first-DL-packet report triggers a Network Service Request and MME provides DL Buffering Duration + Suggested Packet Count, SGW-C includes these in the Sx Report ACK.
