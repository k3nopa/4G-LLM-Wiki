# Source: 3GPP TS 32.299 v16.2.0

**Full title:** Digital cellular telecommunications system (Phase 2+); Universal Mobile Telecommunications System (UMTS); LTE — Telecommunication management; Charging management; Diameter charging applications

**Release:** 16 (2020-11)

**Status:** COMPLETE

---

## Scope (§1)

Specifies the Diameter protocol used for both offline (Rf) and online (Ro) 3GPP charging applications. Covers:
- The logical charging architecture (CTF, CDF, OCF, CGF)
- Offline and online charging scenario requirements
- Diameter protocol aspects for Rf and Ro
- Full AVP dictionary (~250 3GPP-specific AVPs)
- Annex B: Diameter Overload Indication Conveyance (DOIC) for charging

This spec is the bottom-tier (protocol) specification; it is used in conjunction with middle-tier specs (e.g. TS 32.251 for PS domain, TS 32.260 for IMS charging) that define service-specific charging data.

## Chunk ts32299-1 Summary (§4–§5)

### Architecture (§4)
- Logical architecture: CTF → {CDF via Rf (offline), OCF via Ro (online)} → CGF (via Ga) → Billing Domain
- CTF has priority-ordered address list for failover to secondary CDF/OCF
- NEs send charging info only to same-PLMN charging entities
- Roaming online charging: PCEF uses Gy (Ro-based) to HPLMN OCS; TDF uses Gyn
- Single IMSI architecture for EU roaming unbundling (proxy function on Ro)

### Offline Charging Requirements (§5.1)
- Two modes: event-based (single ACR[Event]) and session-based (ACR[Start + Interim* + Stop])
- Charging Data Transfer operation: Request (CTF→CDF) + Response (CDF→CTF)
- Key request fields: Session-Id, Originator-Host/Domain, Operation-Type, Operation-Number, Service-Information

### Online Charging Requirements (§5.2)
- Rating: centralized (OCF) or decentralized (CTF); Unit Determination: centralized (OCF) or decentralized (CTF)
- Centralized UD + Decentralized Rating = NOT supported
- Three cases: **IEC** (immediate debit, no reservation), **ECUR** (reserve + debit, event), **SCUR** (reserve + debit, session)
- IEC → RFC 4006 Direct Debit (one-time event)
- SCUR → RFC 4006 Session-Based Credit-Control; Debit+Reserve combined in single CCR

### Other Requirements (§5.3)
- Re-authorization triggers: idle timeout, threshold, mid-session events
- Termination action: OCF specifies CTF behavior on last-unit consumption
- Account expiration notification

## Chunk ts32299-2 Summary (§6.1–§6.2)

### Diameter Offline Charging Principles (§6.1)
- CTF = Diameter Accounting client; CDF = Diameter Accounting server (RFC 6733)
- CDF implements SERVER STATELESS ACCOUNTING — no ordering constraint on ACR receipt
- ACR record types: START_RECORD, INTERIM_RECORD, STOP_RECORD (sessions), EVENT_RECORD (events)
- ACA[START_RECORD] carries Acct-Interim-Interval (AII) to control INTERIM frequency
- Error cases: CDF connection failure (failover + non-volatile buffer); no-reply retransmission (T-flag); duplicate detection (T-flag; discard if seen, use+mark CDR if original missing); CDF timer-based CDR close

### ACR Message Format (§6.2.2)
- Command-Code 271 REQ PXY; mandatory: Session-Id, Origin-Host/Realm, Destination-Realm, Accounting-Record-Type/Number
- 3GPP adds: Service-Context-Id (Om), Service-Information (Om) — service data from middle-tier TS
- AVPs NOT used in 3GPP: Vendor-Specific-Application-Id, Accounting-Sub/RADIUS/Multi-Session-Id, Accounting-Realtime-Required

### ACA Message Format (§6.2.3)
- Command-Code 271 PXY (no REQ); mandatory: Session-Id, Result-Code, Origin-Host/Realm, Record-Type/Number
- Oc fields: Error-Reporting-Host (proxy that reported non-2001), Acct-Interim-Interval (CDF dictates interval), Failed-AVP

## Chunk ts32299-3 Summary (§6.3–§6.5)

### Ro Protocol Flows (§6.3)
- **IEC:** Single CCR[EVENT_REQUEST, RA=DIRECT_DEBITING, RSU] → CCA[GSU, Cost-Info, Remaining-Balance]; Refund variant: RA=REFUND_ACCOUNT
- **ECUR:** CCR[INITIAL_REQUEST, RSU] → CCA[INITIAL, GSU, VT] → service → CCR[TERMINATION_REQUEST, USU] → CCA[TERMINATION, Cost-Info]
- **SCUR:** CCR[INITIAL, RSU] → CCA[INITIAL, GSU, VT, QHT] → {CCR[UPDATE, USU+RSU] ↔ CCA[UPDATE, GSU, FUI]}* → CCR[TERMINATION, USU] → CCA[TERMINATION, Cost-Info]
- Debit + Reserve **combined** in single CCR[UPDATE] for SCUR — one round-trip per quota cycle

### CCR Message Format (§6.4.2)
- Command-Code 272 REQ PXY; Auth-Application-Id=4
- M: Session-Id, Origin-Host/Realm, Destination-Realm, Auth-Application-Id, Service-Context-Id, CC-Request-Type, CC-Request-Number
- Om: Subscription-Id (Type+Data: IMSI=1/MSISDN=0), Multiple-Services-Indicator, Service-Information
- Oc: Multiple-Services-Credit-Control (MSCC with RSU/USU/Service-Id/Rating-Group/FUI/VT/QHT/QCT/Thresholds/Reporting-Reason/Trigger), Requested-Action (IEC), Termination-Cause, User-Equipment-Info, CC-Correlation-Id, QoS-Information

### CCA Message Format (§6.4.3)
- Command-Code 272 PXY; M: Session-Id, Result-Code, Origin-Host/Realm, Auth-Application-Id, CC-Request-Type, CC-Request-Number
- Oc: MSCC (GSU with CC-Time/CC-Total/Input/Output-Octets/CC-Service-Specific-Units, Tariff-Time-Change, VT, FUI, Result-Code, quota thresholds, QHT, QCT, Reporting-Reason, Trigger, G-S-U-Pool-Reference), Cost-Information, Low-Balance-Indication, Remaining-Balance, Credit-Control-Failure-Handling, Direct-Debiting-Failure-Handling, CC-Session-Failover, OC-Supported-Features/OC-OLR, Redirect-Host

### Key Quota Management Procedures (§6.5)
- **Idle timeout (QHT):** No traffic for QHT seconds → CCR-UPDATE with Reporting-Reason=QUOTA_HOLDING_TIME
- **Validity-Time expiry:** CCR-UPDATE/TERMINATION required before quota expires; Reporting-Reason=VALIDITY_TIME
- **Re-auth triggers:** Trigger/Trigger-Type AVPs (location change, RAT change, QoS change, etc.) → immediate CCR-UPDATE
- **Threshold-based re-auth:** Time/Volume/Unit-Quota-Threshold → CCR-UPDATE before exhaustion
- **Final-Unit-Indication:** FUA=TERMINATE (terminate session) / REDIRECT (redirect to server) / RESTRICT_ACCESS (apply filters)
- **Quota-Consumption-Time (QCT):** Combinational CTP/DTP modes — time consumes volume quota in synchronized periods
- **Envelope reporting:** CTF reports service envelopes per quota period; OCS can request parallel offline envelope records via Offline-Charging AVP
- **CCFH:** TERMINATE / CONTINUE / RETRY_AND_TERMINATE for OCS unreachable scenarios; CC-Session-Failover for alternative OCS
- **AVP bindings (§6.6):** Table 6.6.1.1 offline logical↔ACR/ACA mapping; Table 6.6.2.1 online logical↔CCR/CCA mapping
