# Source: 3GPP TS 23.167 v17.2.0 — IMS Emergency Sessions

**Full title:** Universal Mobile Telecommunications System (UMTS); LTE; IP Multimedia Subsystem (IMS) emergency sessions

## Scope

Defines the **stage 2 service description** for emergency sessions in the IMS (IP Multimedia Core Network Subsystem), covering:
- Emergency session architecture and the additional E-CSCF entity
- Location retrieval function (LRF) and PSAP routing
- Procedures for emergency registration, session establishment, and PSAP interworking
- NG-eCall (vehicle emergency calls with MSD transfer)
- Access-specific annexes: E-UTRAN/LTE, WLAN-to-EPC, HRPD, fixed broadband, roaming

Companion specs: TS 23.228 (IMS general), TS 23.401 (EPS), TS 23.402 (non-3GPP access), TS 23.271 (LCS), TS 23.237 (SRVCC/EATF), TS 23.292 (ICS).

## Key Contributions to Wiki (Chunk 5a-1)

### New entities
- **[E-CSCF](../entities/E-CSCF.md)**: Emergency-CSCF; routes to PSAP via LRF; always in serving network; interfaces: Mw, Ml, Mi, Mg, Mm, Mx, I4, I6
- **[LRF](../entities/LRF.md)**: Location Retrieval Function; consists of RDF + LS; provides ESQK/ESRN/PSAP-URI; Ml to E-CSCF, Sh to HSS, Le to PSAP

### New concept
- **[IMS Emergency Architecture](../concepts/IMS-emergency-architecture.md)**: Full framework — 30 architectural principles, E-CSCF/LRF reference model, UE-detectable vs non-detectable emergency flows, location principles, IP-CAN expectations, NG-eCall introduction

### New reference points
| Point | Between | Purpose |
|---|---|---|
| Ml | E-CSCF ↔ LRF | Location retrieval and PSAP routing info |
| I4 | E-CSCF ↔ EATF | Emergency SRVCC/DRVCC continuity |
| I5 | I-CSCF ↔ EATF | Emergency SRVCC/DRVCC continuity |
| I6 | MSC Server ICS ↔ E-CSCF | CS/IMS interworking for ICS emergency calls |
| Le | LRF ↔ PSAP | Location delivery (NENA E2) — outside 3GPP scope |

### Modifications to existing entities
- **P-CSCF**: detects emergency; selects E-CSCF; queries IP-CAN for location/identifiers; forwards equipment ID for credentialless UEs; informs PCRF of emergency data flow
- **S-CSCF**: modified HSS interaction for emergency registration (bypass barring; UNREGISTERED status handling)
- **HSS**: no barring/restriction for emergency registration IMPU
- **IBCF**: forward emergency sessions; prioritise; invoke AS for attestation signing on operator policy
- **MGCF**: handles NG-eCall as normal emergency call establishment; does not forward initial MSD

## Notable Findings (Chunk 5a-1)

1. **E-CSCF is the only CSCF role always in the serving (visited) network** — even the P-CSCF could theoretically be in the home network for certain configurations, but E-CSCF selection always goes to the visited PLMN. This is architecturally mandated because PSAP routing is geographically determined by serving country regulations.

2. **The ESQK binding pattern** (LRF keeps session record, PSAP queries via Le/E2 using ESQK as key) is a classic indirection mechanism: the IMS network does not expose UE identity or location directly to the PSAP — the PSAP queries the LRF independently. This provides both privacy protection and allows location updates after initial call setup.

3. **Anonymous emergency sessions** are explicitly supported as a first-class case: a UE with no IMS credentials can bypass IMS registration entirely and go directly to P-CSCF → E-CSCF → PSAP, using only the equipment identifier. This is a regulatory requirement (emergency calls must always be possible).

4. **NG-eCall follows all standard IMS emergency procedures** with one addition: the MSD (Minimum Set of Data) is transferred within the emergency session. The architecture does not require a separate bearer — the MSD rides on the same emergency bearer as voice/signalling.
