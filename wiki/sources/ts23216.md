# Source: 3GPP TS 23.216 v16.4.0 (Release 16, 2020-11)

**Full title**: Digital cellular telecommunications system (Phase 2+) (GSM); UMTS; LTE; Single Radio Voice Call Continuity (SRVCC); Stage 2

**Scope**: Architecture enhancements for SRVCC between access systems, for IMS-anchored voice calls:
- E-UTRAN → 3GPP2 1xCS
- E-UTRAN → UTRAN/GERAN (SRVCC and vSRVCC)
- UTRAN(HSPA) → UTRAN/GERAN
- UTRAN/GERAN → E-UTRAN/UTRAN(HSPA) (CS to PS SRVCC)
- NG-RAN → 3GPP UTRAN (5G-SRVCC, introduced this release)

**Key normative references**: TS 23.401 (EPC), TS 23.060 (GPRS), TS 23.237 (IMS Service Continuity), TS 23.292 (ICS), TS 22.278 (EPS requirements), TS 23.502 (5GC procedures), TS 23.501 (5GC architecture)

## Chunk 5b-1 — Concepts and Architecture (§1–§5)

### Key entities introduced

| Entity | Description |
|---|---|
| **1xCS IWS** | 3GPP2 1xCS SRVCC Interworking Solution Function — tunnels 3GPP2 signalling between MME (S102) and 1xRTT MSC (A1) |
| **MSC Server enhanced for SRVCC** | Extended MSC Server with Sv interface; invokes IMS session transfer (STN-SR); coordinates CS HO + IMS transfer |
| **MSC Server enhanced for vSRVCC** | Further extension for video call continuity; handles BS30 bearer reservation |
| **MSC Server enhanced for CS to PS** | Extension for reverse direction; checks subscription + IMS registration + UE capability |
| **MME_SRVCC** | New entity (5G-SRVCC only): bridges N26 (AMF) and Sv (MSC Server) for NG-RAN→UTRAN handover |

### Key reference points introduced

| Reference point | Between | Purpose |
|---|---|---|
| S102 | MME ↔ 1xCS IWS | Tunnel for 3GPP2 1xCS signalling |
| Sv | MME/SGSN ↔ MSC Server | SRVCC handover initiation (also CS to PS) |
| Sv (5G) | MME_SRVCC ↔ MSC Server | 5G-SRVCC variant of Sv |
| N26 (5G-SRVCC extensions) | AMF ↔ MME_SRVCC | 5G-SRVCC HO Indication + STN-SR/C-MSISDN |

### Notable architectural principles extracted

1. **IMS anchor is non-negotiable**: QCI=1 bearer usage restricted to SCC AS-anchored IMS voice sessions (enforced by PCRF).
2. **PS bearer splitting** is the MME/SGSN mechanism that identifies the voice bearer — it relies on QCI=1 uniqueness guaranteed by the PCRF.
3. **STN-SR provisioning path**: HSS → MME via S6a at attach time. MME stores STN-SR and provides it to MSC Server in the Sv message when SRVCC is triggered.
4. **Emergency SRVCC bypasses subscription**: MME/SGSN uses locally configured E-STN-SR in the MSC Server; no STN-SR from HSS required.
5. **5G-SRVCC terminates all PDU sessions**: After NG-RAN→UTRAN HO, the UE is on 3G CS; AMF releases all 5G PDU sessions.
6. **MME_SRVCC is a bridge, not a full MME**: It only stores the N26/Sv binding and converts messages; it does not hold UE MM context.
7. **CS to PS SRVCC eligibility is triple-gated**: UE capability (from IMS per TS 23.237) AND subscription allowance (HSS MAP D per VPLMN) AND active IMS registration.

## Chunk 5b-2 — E-UTRAN and UTRAN(HSPA) Procedures (§6.1–§6.3)

### Key procedures extracted

| Section | Flow | Steps | Key characteristic |
|---|---|---|---|
| §6.1.3 | E-UTRAN → 3GPP2 1xCS | 19 | S102 tunnel relay; 3GPP2 1xCS procedures independent of 3GPP |
| §6.2.2.1 | E-UTRAN → GERAN (no DTM) | 24 | CS-only, Suspend all bearers |
| §6.2.2.1A | E-UTRAN → GERAN/UTRAN (no PS HO) | 24 variant | CS-only, voice deactivated only; PS re-established via RAU |
| §6.2.2.2 | E-UTRAN → UTRAN/GERAN (CS+PS HO) | 19 | Parallel CS+PS paths; two Target Transparent Containers for GERAN |
| §6.2.2.3 | E-UTRAN → UTRAN vSRVCC (PS HO) | 18 | Voice+video transfer; BS30 reservation; SCC AS vSRVCC detection |
| §6.2.2.4 | E-UTRAN → UTRAN vSRVCC (no PS HO) | 18 variant | Video+voice transfer; PS re-established via RAU |
| §6.3.2.1 | UTRAN(HSPA) → GERAN (no DTM) | 24 | SGSN-based; SSD='speech' identification; Gn/Gp vs S4 bearer handling |
| §6.3.2.1A | UTRAN(HSPA) → GERAN/UTRAN (no PS HO) | 24 variant | Voice only deactivated |
| §6.3.2.2 | UTRAN(HSPA) → UTRAN/GERAN (CS+PS HO) | 19 | SRVCC CS KEYS exchange; NONCE IE for GERAN |

### Notable findings

- **Session Transfer (step 10/9) can begin as soon as Prepare HO Response received** (step 8/8b), not waiting for circuit establishment (step 9/8c). Steps 10/11 (Session Transfer/remote update) and step 12 (IMS access leg release) are independent of step 13 (PS to CS Response).
- **SRVCC CS KEYS exchange** (§6.3.2.2 steps 2a/2b): unique to UTRAN(HSPA) → UTRAN. The RNC initiates this before Relocation Required to obtain the CS security keys (Integrity Protection Key + Encryption Key + SRVCC Information) from the SGSN, derived from the PS security context.
- **vSRVCC SCC AS detection logic** (§6.2.2.3 step 10): the SCC AS determines whether to perform vSRVCC vs SRVCC by checking whether the Session Transfer response from the MSC Server includes a video SDP. If video SDP is missing, SCC AS assumes voice-only SRVCC and releases the video bearer from the remote leg — no explicit vSRVCC indicator in SIP.
- **GERAN NONCE IE**: In §6.3.2.2 step 13, when target is GERAN, the source RNC must receive an SRVCC Information IE containing a NONCE IE in the Relocation Command. This NONCE is required for GERAN-side CS security context derivation.

## Chunk 5b-3 — CS-to-PS SRVCC, 5G-SRVCC, Handover Failure, Security (§6.4–§9 + Annex A)

### Key procedures extracted

| Section | Flow | Steps | Key characteristic |
|---|---|---|---|
| §6.4.3.1 | GERAN → E-UTRAN/UTRAN(HSPA) CS-to-PS (no DTM/PS HO) | 16 | MSC retrieves GUTI from UE; Context Request path to old MME/SGSN; CS+PS paths converge at PS bearer setup (step 16) |
| §6.4.3.2 | GERAN+DTM (no DTM HO) / UTRAN (no PS HO) → PS | 16 variant | Old PS node is SGSN in same routing area; no separate PS HO signalling |
| §6.4.3.3 | GERAN/UTRAN with DTM/PS HO → PS | 16 variant | BSC/RNC triggers CS to PS HO Required only when CS to PS SRVCC is supposed to fire |
| §6.5.4 | 5G-SRVCC NG-RAN → 3GPP UTRAN | 18 | AMF → MME_SRVCC (N26 Forward Relocation); MME_SRVCC → MSC Server (Sv per §6.2.2.1A); all PDU sessions released by AMF after CS HO |
| §6.5.5 | 5G-SRVCC Emergency PDU session | — | AMF includes "5G-SRVCC operation possible" in N2; restores SRVCC indication after emergency session release |

### Notable findings

- **CS to PS SRVCC is 16 steps** (vs PS to CS SRVCC at 24 steps for §6.2.2.1). The reverse procedure is structurally simpler because the MSC Server already has the UE in CS context — the complexity is shifted to the MME/SGSN Context Request chain to retrieve the UE's PS bearers from the old node.
- **Step 11a (CS to PS Complete Notification) is the policy trigger**: When dynamic PCC is deployed, receiving this notification causes the PCRF to make policy decisions allowing IMS voice traffic via ATGW on the default bearer. This window (between step 11a and step 16, when the dedicated bearer is ready) requires the default bearer to temporarily carry the IMS voice stream.
- **5G-SRVCC terminates all 5G PDU sessions (§6.5.4 step 17)**: After NG-RAN → UTRAN CS HO, the UE has no PS path. AMF immediately releases all PDU sessions — unlike E-UTRAN SRVCC which may suspend non-voice bearers and resume them after the CS call ends.
- **HO cancellation (§8.1.3) is the most operationally important failure case**: The "Session Transfer started" indication in the PS to CS Cancel Notification ACK (step 5) tells the MME/SGSN that IMS is mid-transfer. This triggers the Session Reestablishment trigger to UE, which must re-INVITE on the existing IMS bearer. This is the correct path to recover call continuity after an aborted SRVCC.
- **Permanent vs temporary failure distinction (§8.1.1a.1)**: The error cause in the PS to CS Response/Complete Notification directly controls whether the MME/SGSN will attempt SRVCC again for this UE. Operators must configure STN-SR correctly — a misconfigured STN-SR generates permanent failure and effectively disables SRVCC for that subscriber.
- **NCL algorithm (Annex A) is the operational gating mechanism**: Without the "SRVCC operation possible" indicator set to true by MME/eNB, the E-UTRAN will not include VoIP-incapable cells in the NCL. This means SRVCC can only be attempted when both UE and network signalled readiness. The QCI=1 bearer existence is a second gate — confirming there is actually a voice session to transfer.
