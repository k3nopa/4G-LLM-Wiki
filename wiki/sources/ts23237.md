## Chunk ts23237-1 Summary — §3–§5

**Pages written:**
- `wiki/concepts/IMS-service-continuity.md` — new: full IMS SC concept (Access Transfer + IUT), 3pcc model, Collaborative Session, identifiers (STN/STI/C-MSISDN/IMRN), signalling/bearer path table
- `wiki/entities/SCC-AS.md` — new: SCC AS entity — functions (Access Transfer 3pcc, C-MSISDN, STN-SR, Session State Information, IUT, T-ADS), interfaces (ISC/Sh/OMA DM)
- `wiki/entities/ATCF.md` — new: ATCF entity — role, STN-SR allocation, ATGW control, AT execution, STI-rSR for CS-to-PS, interfaces (Mw/Mx/Iq/Ix/I2)
- `wiki/entities/ATGW.md` — new: ATGW entity — media anchor, transcoding, Iq/Ix interfaces, IMS-AGW or TrGW mapping
- `wiki/interfaces/IMS-reference-points.md` — updated: added Iq, Ix, I2 reference points

**Key findings:**
- ATCF is in **serving** network; SCC AS is in **home** network — this separation enables SRVCC without home round-trip
- STN-SR in HSS points to ATCF (not SCC AS) when ATCF is deployed; this is the routing hook for SRVCC requests
- ATU-STI is the token SCC AS gives ATCF at registration time; ATCF uses it to notify SCC AS after AT completes
- 3pcc at SCC AS means two SIP dialogs: Access Leg (UE↔SCC AS) + Remote Leg (SCC AS↔remote party)
- ATGW may be TrGW (IBCF co-location) or IMS-AGW (P-CSCF co-location)
- The Controllee UE is deliberately unaware of its role — any IMS UE can be a Controllee UE

## Chunk ts23237-3 Summary — §6.3.1–§6.3.2

**Pages written:**
- `wiki/procedures/PS-CS-access-transfer.md` — new: full AT flow catalogue (PS→CS/CS→PS Dual Radio, SRVCC, CS→PS with ATCF, PS-PS full/partial/early-dialog, PS-PS+PS-CS conjunction), plus Remote Leg Update and Source Access Leg Release sub-procedures

**Key findings:**
- **CS→PS Single Radio with ATCF (§6.3.2.1.10)** is the only flow where ATCF acts autonomously first (Session Transfer Notification → ATGW switch → UE Session Transfer Complete) before contacting SCC AS — this is the reverse-SRVCC fast path that avoids home round-trip latency
- **Session State Information (SSI)**: after an SRVCC, SCC AS sends SSI to MSC Server containing STN-SR for each held session; MSC Server can initiate additional CS calls for those sessions — this is the mechanism for transferring held sessions during SRVCC
- **Session selection algorithm**: most-recently-active session → transferred first; second-most-recently-active → held (SSI); all others → released
- **PS-PS partial transfer (§6.3.2.2.2)** leaves the old Access Leg alive for the remaining media subset — two simultaneous IP-CAN legs coexist intentionally
- **PS-PS in conjunction with PS-CS (§6.3.2.3)** requires two separate INVITE flows (one STI per domain) for non-ICS UEs; ICS UEs unify via PSI DN correlation at SCC AS
- **ATCF MUST NOT modify the dynamic STI** (confirmed again in AT flows) — it passes through transparently; only SCC AS interprets it
- Source Access Leg Release for SRVCC is conditional on Gm retention — if Gm is not retained (common in basic SRVCC), the source PS leg was already cleared by radio procedures

## Chunk ts23237-4 Summary — §6.3.3–§6.5

**Pages written:**
- `wiki/procedures/IMS-SC-media-and-services.md` — new: 12 media-adding/deleting sub-cases (local/remote, CS+PS and PS-only), ICS mid-call fallback (§6.3.5), operator policy structure with `shall/should/may` semantics, supplementary services impact table (22 services), detailed rules for 6 impacted services (CDIV, HOLD, CONF, ECT, CW, MCID)

**Key findings:**
- **§6.3.3 is T-ADS in action post-establishment**: when UE-2 adds media to an existing CS-only session, T-ADS in SCC AS decides the delivery path (PS), uses C-MSISDN for correlation, splits the session, and ensures the new media is only delivered to UE-1 (not other UEs of the same subscriber). The same T-ADS logic that applies at session setup applies to media modification events.
- **§6.3.5 fallback to MSC Server mid-call feature**: if ICS UE loses Gm reachability, SCC AS falls back to MSC Server assisted mid-call — UE then uses TS 24.008 (CS NAS) for service control instead of SIP. This is the last-resort path for ICS service control continuity.
- **Operator policy `shall/should/may` is the AT trigger control knob**: operators can mandate automatic AT on preferred access availability (`shall`), recommend with UE discretion (`should`), or leave entirely to user preferences (`may`). All three levels must still be reconciled against Local Operating Environment Information (LOEI: radio quality, QoS, power, application requirements).
- **Most supplementary services are not impacted by split Access Legs**: CDIV, HOLD, CONF, ECT, CW, MCID require adaptation — in all cases, the rule is that the UE may act on any Access Leg, and the SCC AS propagates to all affected legs or selects the appropriate one.
- **HOLD is the most symmetric**: both local and remote HOLD must propagate across all Access Legs containing the affected media — SCC AS is responsible for fan-out.

## Chunk ts23237-6 Summary — §6c–§6d, §7–§8, Annexes A–C

**Pages written:**
- `wiki/procedures/IMS-IUT-procedures.md` — updated: appended §6a.8.3.3 (cross-subscription Session Discovery), §6a.9 (target-initiated IUT: 5 variants same/different subscription including Solicit IUT), §6a.10 (Media Flow Replication via MRF: push/pull × same/different subscription), §6a.11 (Session Replication: 4 variants — creates independent sessions, no Collaborative Session), §6a.12 (User Authorization and Preferences: SCC AS 3-layer authorization, UE-based, Ut interface configurability)
- `wiki/procedures/SRVCC-emergency-procedures.md` — new: §6c SRVCC emergency (origination anchor at EATF, active+early-dialog AT with single/multiple EATF instances via forking/redirection, Normal/Limited Service Mode) + §6d DRVCC emergency for WLAN (E-STN-DR, make-before-break)
- `wiki/concepts/IMS-SC-charging.md` — new: §7 security (no additional requirements beyond TS 33.102/33.203) + §8 charging strategy (cohesive SCC AS CDR, CS/IMS double-billing prevention, subsequent Access Leg treatment, online charging IMS-only, ICID roaming correlation) + §8.2 accounting
- `wiki/procedures/SRVCC-enhancements.md` — new: Annex B codec re-negotiation after SRVCC (B.2.1), codec inquiry prior to SRVCC using PS-to-CS-Preparation-Request/Response (B.2.2), reject SRVCC on call state incompatibility (B.3)
- `wiki/entities/ATCF.md` — updated: appended Annex C (ATCF in HPLMN without IMS-level roaming interfaces — Mx via IBCF chain, PS→CS AT flow without ATGW media anchor)

**Key findings:**
- **EATF anchors emergency session as routing B2BUA** (§6c.1): exactly mirrors what ATCF does for normal sessions (§6.3.1.3); E-STN-SR is allocated at origination and returned to UE in the response — SRVCC readiness established before any emergency occurs
- **Multiple EATF instances (§6c.2.3–2.4) use forking or redirection**: forking sends one INVITE per EATF instance; the matching instance (equipment identifier match) responds 200 OK and the others reject; redirection has EATF-1 return 3xx with backup EATF-2 contact
- **Early-dialog SRVCC uses UPDATE, not Re-INVITE** (§6c.2.2): because the SIP dialog is in early state; also sends Session State Information to MSC Server so it can align CS call state with IMS pre-alerting/alerting state
- **DRVCC is make-before-break** (§6d.2): WLAN emergency call remains active while CS leg is being set up — emergency call maintained throughout; uses normal CS call setup (not CS emergency setup), so NO higher RRC priority
- **SC charging is IMS-only for online charging**: CS prepaid logic MUST NOT be invoked for anchored CS origination/termination; SCC AS is sole OCS reporting point and must report both initial session and Session Transfer info
- **Codec inquiry (Annex B.2.2) decouples codec selection from radio reservation**: MSC Server queries ATCF BEFORE committing to radio procedure (PS-to-CS-Preparation-Request with MSC Supported Codec List); ATCF returns IMS Preferred Codec List; enables transcoding-free path selection
- **Annex C ATCF deployment (HPLMN, no IMS roaming)**: ATCF in HPLMN connected via Mx through IBCF chain; no ATGW media anchor → ATCF acts as relay for Access Transfer message to SCC AS; mid-call feature success depends on SIP header fidelity across IBCF chain

## Chunk ts23237-5 Summary — §6a

**Pages written:**
- `wiki/procedures/IMS-IUT-procedures.md` — new: complete IUT procedures: Collaborative Session establishment (4 variants), media transfer/add/delete/modify (Controller and Controllee initiated, same and different subscriptions), Collaborative Session Control transfer (voluntary push/pull/loss), session release (Controller and remote initiated), IUT without Collaborative Session, supplementary services for IUT, target discovery, session discovery

**Key findings:**
- **SCC AS establishes Controllee Access Leg BEFORE engaging remote party at originating setup (§6a.2.3)**: Required to obtain full SDP offer for Controllee media — without UE-2's SDP, SCC AS cannot send a complete offer to the remote party. Non-obvious ordering constraint with no equivalent in standard origination.
- **Controllee-initiated actions always require Controller authorization (§6a.12)**: Controllee can initiate transfers (§6a.3.3a), releases (§6a.4.5), modifications (§6a.4.6), and session control pulls (§6a.4a.5) — but each routes through SCC AS to Controller UE (or SCC AS acts on Controller's behalf if pre-configured). Controllee is not autonomous.
- **SCC AS-2 acts as routing relay for cross-subscription IUT**: when Controllee UE belongs to different subscription than Controller, SCC AS-2 notes the Collaborative Session is anchored at SCC AS-1 and routes all future requests for that session back to SCC AS-1. Key mechanism enabling cross-subscription Collaborative Sessions.
- **Control loss preference decouples Controller UE availability from session continuity**: SCC AS autonomously transfers Collaborative Session Control to a Controllee upon loss of Controller — no round-trip to Controller needed. If preference is not configured, entire Collaborative Session is released. Only case where SCC AS acts autonomously on session control.
- **IUT without Collaborative Session (§6a.6) transfers entire session atomically — target UE does NOT need IUT capabilities (§6a.6.1)**: target UE receives a normal IMS session setup. IUT capability only required for partial media transfer (Collaborative Session).
- **Session Discovery (§6a.8.3) is the prerequisite for target-initiated IUT**: before UE-2 can request to take over UE-1's session, it must discover session ID and media flow identifiers via Session Discovery. SCC AS authorizes/filters before disclosing session info.
