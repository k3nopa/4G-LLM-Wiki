# IMS CDR Field Reference — Per-Node Content

This page documents §6.1.3 of TS 32.260: the content and structure of IMS Charging Data Records (CDRs) as generated by the CDF at the Bi interface. Twelve CDR types are defined, one per IMS node type.

Detailed ASN.1 field encoding is in TS 32.298. The IMS Information AVP structure (used in Rf/Ro Service Information) is documented in §6.3 — see [IMS converged charging flows](../procedures/IMS-converged-charging-flows.md) for the message-level overview.

---

## §6.1.3.1 CDR Types

| CDR Type | Generating Node | CDR Mode |
|---|---|---|
| S-CSCF-CDR | S-CSCF | S/I/S/E |
| P-CSCF-CDR | P-CSCF | S/I/S/E |
| I-CSCF-CDR | I-CSCF | **E only** |
| MRFC-CDR | MRFC | **S/I/S only** |
| MGCF-CDR | MGCF | S/I/S/E |
| BGCF-CDR | BGCF | **E only** |
| AS-CDR | SIP AS | S/I/S/E |
| IBCF-CDR | IBCF | S/I/S/E |
| E-CSCF-CDR | E-CSCF | S/I/S/E |
| TRF-CDR | TRF (Transit Routing Function) | S/I/S/E |
| ATCF-CDR | ATCF (Access Transfer Control Function) | S/I/S/E |
| TF-CDR | IMS TF (Transit Function) | S/I/S/E |

**CDR modes:** S = Start, I = Interim, S = Stop, E = Event. I-CSCF and BGCF are event-only (never in session signalling path after initial routing). MRFC is session-only (conference sessions, no standalone events).

## §6.1.3.2 CDR Triggers at the CDF

### Session-Related CDRs (§6.1.3.2.1)

- CDF **opens** a session CDR upon receipt of `Charging Data Request [Start]`
- CDF **updates** (partial CDR) upon receipt of `Charging Data Request [Interim]` — triggered by session modification (RE-INVITE/UPDATE) or Charging Data Interim Interval expiry
- CDF **closes** upon receipt of `Charging Data Request [Stop]`
- Unsuccessful setup: CTF sends `Charging Data Request [Event]` (not Start); CDF creates event-only record

### Session-Unrelated CDRs (§6.1.3.2.2)

- CDF generates a CDR for each received `Charging Data Request [Event]`
- "One-off" action — one CDR per Event message
- Creation of partial CDRs does not apply to session-unrelated CDRs

## Per-Node CDR Distinctive Features

### §6.1.3.3 S-CSCF CDR

**CDR mode:** S/I/S/E

The S-CSCF CDR is the most comprehensive IMS CDR. Distinctive fields:

| Field | Description |
|---|---|
| Application Servers Information | Grouped field: "Application Servers Involved" (AS SIP URIs) + "Application Provided Called Parties" (addresses determined by AS, e.g. SIP URI E.164) + Status |
| Transit IOI List | Transit IOI values from P-Charging-Vector; each occurrence = transit IOI from path inbound or outbound from S-CSCF |
| Number Portability routing information | DNS/ENUM result from S-CSCF in calling user's home network |
| Carrier Select routing information | Carrier select result from S-CSCF |
| NNI Information | Roaming LBO NNI info; present only when VPLMN routing is applied — NNI Type indicates loopback decision |
| SIP Route header received | Topmost route header in received initial INVITE (for requests from the served user prior to triggering ASes) |
| SIP Route header transmitted | Route header in transmitted INVITE (for requests toward/from served user when VPLMN routing is applied) |

### §6.1.3.4 P-CSCF CDR

**CDR mode:** S/I/S/E

Unique to P-CSCF:

| Field | Description |
|---|---|
| Served Party IP Address | IP address of calling or called party (depending on P-CSCF role: originating vs terminating) |
| Local GW Inserted Indication | OMR: whether local IMS-AGW is inserted for the media component |
| IP realm Default Indication | OMR: whether User Plane IP realm is the Default IP realm |
| IMS Application Reference ID (OC) | IARI from P-Asserted-Service header in SIP request; only present if CDR triggered on SIP REGISTER 200 OK _(FFS for extraction mechanism per Editor's Note)_ |
| Related IMS Charging Identifier | Related ICID for access transfer cases |
| Related IMS Charging Identifier Generation Node | Identifier of server that generated the Related ICID |

P-CSCF CDR: only one address in List Of Calling Party Address (unlike S-CSCF which can list multiple).

### §6.1.3.5 I-CSCF CDR

**CDR mode: E only** (event)

The I-CSCF is event-only in charging — it routes the initial INVITE and exits the path. Distinctive fields:

| Field | Description |
|---|---|
| S-CSCF Information | Info relating to S-CSCF: server capabilities (at registration) OR S-CSCF address on session establishment event |

I-CSCF CDR omits: SDP media components, Service Delivery End Time Stamp, Record Opening Time, Application Servers Information, Transit IOI List, NNI Information, SIP Route headers. Includes: List Of Associated URI, Called Party Address, Carrier/Number Portability (OC).

### §6.1.3.6 MRFC CDR

**CDR mode: S/I/S only** (no event CDR)

The MRFC charges for conference resource usage. Distinctive fields:

| Field | Description |
|---|---|
| Session ID | SIP Call-ID of the **conference session** (hosted by MRFC) |
| Requested Party Address | For SIP transactions — originally posted party; if B2BUA, the conference originator |
| Online Charging Flag | Present but NOTE: "No proof that online charging action has been taken" (MRFC can only signal it saw the flag, not that OCS acted) |

MRFC CDR omits: Number/Carrier portability, Private User ID, Transit IOI List, NNI Information.

### §6.1.3.7 MGCF CDR

**CDR mode:** S/I/S/E

PSTN-specific distinctive fields:

| Field | Description |
|---|---|
| ISUP Cause | When session released via ISUP: reason the call was released |
| Trunk Group ID Incoming/Outgoing | Outgoing trunk group ID for outgoing session/call; incoming trunk group for incoming |
| Bearer Service | Used bearer service for the PSTN leg |

MGCF CDR omits: Access Correlation ID (PSTN has no IP-CAN bearer), Application Servers Information, Private User ID, List of Subscription Id.

### §6.1.3.8 BGCF CDR

**CDR mode: E only** (event)

The BGCF makes a single routing decision and exits the path. Simplified CDR:

| Field | Description |
|---|---|
| NNI Information | Roaming LBO NNI info; NNI Type indicates BGCF performed the loopback decision |

BGCF CDR omits: SDP components, Service Delivery End Time Stamp, Record Opening Time, Application Servers Information, Subscription Id, Private User ID, Transit IOI List.

### §6.1.3.9 SIP AS CDR

**CDR mode:** S/I/S/E

The AS CDR is the most distinctive due to B2BUA/SCC AS support:

| Field | Description |
|---|---|
| Session ID | When B2BUA: **incoming** dialog Call-ID; for OneChargingSession: contains either incoming or outgoing Call-ID involved during IMS session setup |
| Outgoing Session ID | When B2BUA: **outgoing** dialog Call-ID (SIP Call-ID per RFC 3261); **NOT** present when OneChargingSession option is used |
| Alternate Charged Party Address | Address of alternate party identified by AS at session initiation; charged instead of calling party (§5.2.2.1.14 alternate charging) |
| List of Requested Party Address | List of Requested Party Address (may differ from Called Party) |
| Service Specific Info | AS-provided service-specific data: Service Specific Data (OM) + Service Specific Type (OM) |
| VLR Number | VLR Number from UDA response (TS 29.328) — for SRVCC/inter-UE transfer scenarios |
| MSC Address | MSC Number from UDA — for SRVCC/inter-UE transfer |
| List of Access Transfer Information | Grouped list of subsequent session transfers (SCC AS): Access Transfer Type, Inter-UE Transfer flag, Access Network Information (from transfer INVITE), Subscriber Equipment Number, Instance Id, Related IMS Charging Identifier, Access Transfer Time |
| Initial IMS Charging Identifier | ICID of initial session created for IMS service continuity; **not used** for OneChargingSession option |
| TAD Identifier | Type of access network (CS or PS) through which the session shall be terminated |
| 3GPP PS Data Off Status | PS Data Off Status at registration (from TS 23.228) |

### §6.1.3.10 IBCF CDR

**CDR mode:** S/I/S/E

IBCF CDR is the inter-network border charging record. Distinctive fields:

| Field | Description |
|---|---|
| Transcoder Inserted Indication | Whether a transcoder is inserted for the media component in SDP — present in SDP Media Components sub-group (OMR indicator, not present in S-CSCF or MRFC CDRs) |
| NNI Information | Grouped field that **may occur more than once** (e.g. when routing capability in support of transit is collocated with the IBCF). Contains: |
| — Session Direction | Inbound or outbound service request on control plane for interconnection/roaming |
| — NNI Type | Non-roaming / roaming with loopback / roaming without loopback |
| — Relationship Mode | Whether the other functional entity (neighbouring network) is regarded as same trust domain |
| — Neighbour Node Address | Control plane IP of neighbouring network contact point for interconnection/roaming |
| IMS Communication Service ID | Special rule: present when IBCF is **downstream** from S-CSCF serving Orig party ("+g.3gpp.ics-ref" header in SIP request) OR **upstream** from S-CSCF serving Orig party (P-Asserted-Service in SIP response) |

IBCF CDR includes all three OMR indicators (Local GW Inserted, IP realm Default, Transcoder Inserted) in SDP media components.

### §6.1.3.11 E-CSCF CDR

**CDR mode:** S/I/S/E

E-CSCF CDR is similar to S-CSCF but for emergency sessions. Distinctive aspects:

- **Called Party Address**: can be SIP URI, TEL URI, or **URN** (emergency service URN per TS 23.167) 
- **Requested Party Address**: similarly can be URI or URN
- Includes Application Servers Information (LRF, PSAP routing ASes)
- No SIP Route header fields, no NNI Information, no Transit IOI List

### §6.1.3.12 TRF CDR

**CDR mode:** S/I/S/E

TRF (Transit Routing Function) handles visited network routing in LBO roaming. Distinctive:

| Field | Description |
|---|---|
| NNI Information | **May occur more than once** (e.g. when routing capability in support of transit is collocated with TRF). NNI Type: non-roaming vs roaming with loopback vs roaming without loopback |
| Transit IOI List | May occur several times; each occurrence = transit IOI values from path inbound to or outbound from the TRF |
| Application Servers Information | Present (for TRF acting as transit network providing IMS application services) |

### §6.1.3.13 ATCF CDR

**CDR mode:** S/I/S/E

ATCF (Access Transfer Control Function) anchors media in the visited network for PS↔CS access transfer. Distinctive fields:

| Field | Description |
|---|---|
| Session ID | When B2BUA: incoming session Call-ID; for OneChargingSession: either incoming or outgoing Call-ID |
| Outgoing Session ID | When B2BUA: outgoing session Call-ID; not used for OneChargingSession |
| List Of Calling Party Address | Includes Correlation MSISDN |
| List of Access Transfer Information | Nested in: Access Network Info Change (OC) — describes subsequent session transfers via ATCF: Access Transfer Type, Access Network Information (from transfer SIP INVITE), Subscriber Equipment Number, Instance Id |
| Related IMS Charging Identifier | Related ICID for access transfer |
| Initial IMS Charging Identifier | ICID of initial SIP session created for IMS service continuity; not used for OneChargingSession |
| NNI Information | Present only if RAVEL "VPLMN routing" applied; NNI Type indicates loopback |

> **Note:** When ATCF is collocated with P-CSCF or IBCF, combined offline charging applies with applicable fields from Table 6.3.2.1 (§6.3.2).

### §6.1.3.14 TF CDR (IMS Transit Function)

**CDR mode:** S/I/S/E

TF (IMS Transit Function) is the transit routing node within operator IMS cores. Distinctive:

| Field | Description |
|---|---|
| Transit IOI List | May occur several times; each occurrence = transit IOI values from path inbound to or outbound from the TF; **includes the TF's own Transit IOI** |
| Application Servers Information | Covers the case of the transit network providing IMS application services |
| NNI Information | NNI Type indicates loopback routing; "S-CSCF performed the loopback decision" |

TF CDR omits: Access Network Information, Additional Access Network Information, Cellular Network Information (all "Not in TF"). Editor's Note flags need for further investigation on parameter occurrence rules.

## §6.1.3 CDR Cross-Reference: IMS Information vs CDR Fields

The CDR fields at the Bi interface directly correspond to sub-fields of the **IMS Information** AVP (§6.3.1.2) carried in the Rf Charging Data Request. The CDR is constructed by the CDF from the IMS Information received. Key mappings:

- `Session ID` CDR field ← `User Session ID` in IMS Information
- `Calling Party Address` CDR ← `Calling Party Address` in IMS Information  
- `Called Party Address` CDR ← `Called Party Address` in IMS Information
- `IMS Charging Identifier` CDR ← `IMS Charging Identifier` in IMS Information (ICID)
- `List of SDP Media Components` CDR ← `SDP Media Component` in IMS Information
- `Application Servers Information` CDR ← `Application Server Information` in IMS Information
- `NNI Information` CDR ← `NNI Information` in IMS Information

The full IMS Information structure and per-node field applicability matrix (Table 6.3.2.1) is documented in §6.3 of TS 32.260 — covered in chunk ts32260-6.
